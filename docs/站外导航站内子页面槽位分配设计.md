# 站外导航站内子页面槽位分配设计

这份文档只写设计口径，不放算法实现代码。

对应算法代码单独放在：

- [站外导航站内子页面分配算法](./站外导航站内子页面分配算法)

## 1. 目标

当当前页面里某个站点同时出现多个可导航子页面时，外站导航场景不再直接让所有子页面和别的站点平铺竞争。

正确流程是：

1. 先按站点比较站点级权重
2. 再给每个站点分配子页面预加载槽位
3. 最后站点内部的子页面再按页级权重争抢这些槽位

## 2. 输入对象

### 2.0 组内站点选择上限

当前这层设计不再只有一个统一的 `siteSelectionLimit`。

外站场景固定拆成两组：

1. 原生预加载组
   - 使用 `siteSelectionLimit`
2. 真实标签页预加载组
   - 使用 `tabSiteSelectionLimit`

对任意一组来说：

- 该组自己的站点选择上限，表示本轮最多允许多少个站点进入“已选站点集合”
- 它和该组自己的页面预加载槽位数是两套独立上限

也就是说，当前外站场景固定拆成两组，每一组内部都有两种不同的 cap：

1. 该组的站点选择上限
2. 该组的页面预加载槽位上限

前者决定：

- 最终有多少个站点能进入本轮分配

后者决定：

- 这些已选站点一共如何分配页面级预加载槽位

### 2.1 `scores`

- 表示当前轮参与分配的站点级权重
- 每个值都已经是站点级频数、站点级 AI 关键词匹配等信号合成后的结果

### 2.2 `caps`

- 表示每个站点在当前页面里真实存在的“可导航子页面数量上限”
- 这个值不是配置，不是 flag，而是对当前页面候选链接检测后的结果

例如：

- GitHub 在当前 Google 搜索页里出现了：
  - 主页
  - Login
  - Explore
- 那么 GitHub 的 `cap` 就是 `3`

如果某个站点在当前页面只出现了 1 个可导航子页面，那么它最多只能拿到 1 个槽位。

### 2.3 `a`

- 表示当前这一轮总共可分配的子页面槽位数
- 它只在“已选中的站点集合”内部起作用
- 不再对页面上所有站点直接分配

## 3. 新算法相对旧算法的关键变化

旧方案里更像是：

- 按权重比例分配
- 再用约束修正

你现在这版算法新增了一个很关键的前提：

- 只要一个站点被纳入本轮分配集合，它至少先拿到 1 个基础槽位

然后再把剩余槽位按权重做二次优化分配。

也就是：

1. 先给每个已选站点保底 `1`
2. 再把剩余 `a - n` 个槽位，按变换后的权重重新分配
3. 每个站点额外可拿的数量受 `cap - 1` 限制

这版算法的意义是：

- 避免低权重但已入选的站点被直接分到 `0`
- 保证本轮被选中的站点至少有一个真实子页面能参与预加载
- 再让高权重站点在额外槽位上获得优势

## 4. 这版算法的隐含前提

这版算法有一个必须明确的前提：

- 输入的站点集合已经是“本轮已选中的站点集合”

因为它要求：

- `a >= n`

其中：

- `a` 是总槽位数
- `n` 是参与分配的站点数

所以如果：

- 页面上出现了 12 个站点
- 但当前轮只允许分配 5 个总槽位

那就不能直接把这 12 个站点全送进这条算法。

必须先做一个上游步骤：

- 先选出本轮真正参与分配的站点集合

然后再对这个“已选站点集合”使用这条算法。

## 5. 当前推荐的上游流程

当前外站场景建议固定成：

1. 当前页面候选链接按站点聚类
2. 每个站点先计算 `siteWeight`
3. 先按 `siteWeight` 排序
4. 用当前分组自己的站点选择上限对站点列表做截断，得到本轮“已选站点集合”
5. 只对这个集合统计每个站点的真实 `cap`
6. 再用 `allocateSelectedWithMinOne(...)` 给这些已选站点分配页面槽位
7. 每个站点内部再按子页面级权重排序，抢本轮拿到的槽位

也就是说，这条算法不是“站点筛选 + 站点分槽”二合一。

它只负责：

- 对“已经选中的站点集合”做整数槽位分配

站点筛选本身应单独发生在它之前，并且明确受当前分组自己的站点选择上限控制。

## 6. `transform` 的意义

当前算法增加了：

- `transform = x => Math.sqrt(x)`

这一层的作用是：

- 先压缩站点权重之间的差距
- 避免特别高的站点权重把额外槽位全部吃光

也就是说：

- 基础保底负责“广度”
- `transform` 后的加权负责“强站点拿更多额外槽位”

默认 `sqrt` 是合理的中间方案：

- 会保留强弱差异
- 但不会像线性权重那样差距过大

后面如果测试发现：

- 高权重站点仍然拿得太少
  - 可以把 `sqrt` 改得更接近线性
- 高权重站点吃得太多
  - 可以换成更强压缩的变换

## 7. 这版算法和频数系统的关系

这条算法只负责：

- “站点之间如何分配子页面槽位”

它不直接决定：

- 站点级权重怎么计算
- 子页面级权重怎么计算

当前仍然保持：

- 站点级权重 = 站点频数信号 × 站点级 AI 匹配
- 子页面级权重 = 外站到站内子页面频数信号 × 子页面级 AI 匹配

所以算法位置应该是：

`siteWeight 排序之后、站点内部 pageWeight 排序之前`

更完整地说，当前外站场景应理解成：

`按执行策略分组 -> 组内站点聚类 -> siteWeight 排序 -> 组内站点上限截断 -> allocateSelectedWithMinOne(a, scores, caps) -> 站点内部 pageWeight 排序`

## 8. 原有“默认子页面继承站点值”机制继续废弃

在这版设计里：

- 站点值先在站点层发挥作用
- 然后通过分槽进入站点内部

因此不再需要：

- 把站点值直接继承给某个默认子页面

这条旧机制现在应继续保持废弃状态，不应再作为 fallback 放回去。
