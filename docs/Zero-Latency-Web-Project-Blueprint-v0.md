# Zero-Latency Web 项目蓝图 v0

_Zero-Latency Web Project Blueprint v0_

## 1. 项目核心愿景 / Core Vision

这个项目的根目标，不是做一个普通的“网页加速插件”，而是做一套更智能、更激进的预加载系统，让 Web 网页在高频使用场景下获得“犹如桌面软件般丝滑的无加载体验”。

抓住的不是 benchmark 上快一点，而是用户体感上的那种“点击马上就能打开”的感觉。对经常网页办公的人来说，这种体验的核心不是指标优化，而是“网页终于不再像网页”。

从定位上看，这不是一次小修小补，而是在尝试把浏览器网页交互往“桌面软件体验”那个方向推一大步。项目代号继续沿用 `Zero-Latency Web`。

## 2. 产品形态与总体架构 / Product Form and Overall Architecture

项目明确放弃了“纯前端插件自己想办法硬做完一切”的路线，转而采用桌面软件 + 浏览器插件的混合架构。问题的本质不是普通 Chrome 扩展能优雅解决的，因此整体结构分为以下几层：

- 控制中枢：本地桌面软件，作为系统“大脑”。
- 浏览器触手：Chrome 插件，作为浏览器内的感知与执行层。
- 通信机制：通过无需拉调试的原生通道协同。
- 沙盒容器：同时存在“两个 Chrome 页面/窗口”，一个是用户实际使用的前台窗口，另一个专门承载已预加载好的隐藏标签页。

这不是附着在浏览器上的一个小功能，而是一套由本地高权限控制层、浏览器内行为感知层、隐藏承载窗口，以及页面置换逻辑共同组成的系统。

## 3. 预加载引擎的核心思想 / Core Philosophy of the Preloading Engine

### 3.1 硬件遥测与动态阈值 / Hardware Telemetry and Dynamic Thresholds

预加载系统不是一套写死的固定策略，而是从一开始就依赖电脑配置与系统当前状态来动态决策。系统首先会检测：

- 用户电脑配置
- 当前 CPU 占用
- 当前内存占用
- 其他可感知的资源压力指标

然后根据这些状态，自动决定：

- 哪些预加载手段启用
- 哪些预加载手段禁用
- 触发阈值要多细才触发
- 一次预加载的范围有多大

这意味着系统不是“大家都吃同一套激进逻辑”，而是根据机器性能和实时负载做自适应。

### 3.2 基于点击链的马尔可夫行为树 / Markov Behavior Tree Based on Click Chains

系统的核心算法直觉之一，是在后台基于用户点击链建立行为树。随着用户长期使用，点击链会逐渐形成“粗壮主干”和“不断细化的分支”。系统的重点不是对所有分支都暴力预载，而是优先盯住那些被反复验证过的“主干路径”。

原始设想如下：

- 用户无数次点击，会让行为树逐渐长出主干与分支。
- 当用户在主干路径上前进时，预加载系统就直接把主干后续部分加载好并缓存下来。

这意味着系统并不是在做“随机猜测”，而是在做以长期行为统计为基础的高概率路径预热。

### 3.3 高配机器的闲时唤醒 / Idle-Time Wakeup on High-End Machines

对高配机器开放更激进的策略：

- 当用户在当前页面失焦
- 或者长时间停顿
- 且机器配置足够强

系统可以开始唤醒其他原本休眠的标签页，提前准备点击链后续网页。

核心逻辑很直接：用户不动的时候，也是系统抢跑的时候。

### 3.4 极限模式：矢量靠近预测 / Extreme Mode: Proximity-Based Prediction

更高级、更极端的模式是：

- 只要用户的鼠标出现“靠近某链接、某按钮、某目标区域”的行为趋势，就立刻开始预加载。

对此的判断很明确：

- 这是最极端、最吃性能的模式。
- 还可能造成服务器额外负担。
- 不应该默认开启，而应该作为激进模式或实验模式存在。

这部分体现的是项目上限意识：愿意把技术上限想得很高，但默认策略必须克制。

## 4. 状态连贯与登录态突破 / State Continuity and Login-State Breakthrough

项目不满足于只做“匿名页面预载”，而是进一步考虑登录态网页如何预加载。

一种激进设想是：

- 如果密码管理器能够直接填入密码
- 那么预加载器就可以在后台自动完成相关流程
- 从而跳过“加载 -> 密码管理器填入 -> 用户点击确认”
- 直接推进到登录后的页面

这部分本质上是对“状态连贯”的追求：预加载不只是把页面壳子提早打开，而是把用户真正想去的最终状态提前准备出来。

在当前总体设想里，这类功能依然存在，但已被明确降级为：

- 很酷
- 很危险
- 用来撑上限
- 默认不开
- 必须强警告
- 不能影响项目整体声誉

也就是说，它是上限展示层，而不是主项目定义层。

## 5. 视觉层的零延迟：跨窗口标签页置换 / Visual Zero-Latency via Cross-Window Tab Replacement

为了做到肉眼层面的“瞬开”，项目设计了一个关键机制：跨窗口标签页置换。

基本构想是：

- 用户进行看似正常的跳转，例如点击一个本该去登录页或下一页的链接。
- 插件直接拦截这次点击。
- 然后把后台隐藏窗口里已经加载好的那个标签页，瞬间加入到用户当前正在使用的窗口中。

体感预期是：

- 用户会觉得和正常打开网站几乎没有区别。
- 唯一可能的细微差别，只是也许没有传统意义上的网址点击反馈。

这就是整个项目最显眼的“魔术时刻”之一：它把后台预热结果，通过窗口级切换，直接变成前台的“无加载感”。

## 6. UX 痛点与工程妥协：后退按钮问题 / UX Pain Point and Engineering Compromise: The Back Button

项目没有回避这个方案的最大 UX 问题：由于做了标签页置换，Chrome 可能会认为当前页面没有正常历史栈，于是后退按钮会变灰，不能正常后退。

围绕这个问题，已经形成过几轮设想。

### 6.1 阶段一：历史嫁接 / History Grafting

最初的思路是：

- 自己维护一套历史记录
- 然后把上一个页面的历史数据“嫁接”到当前页面
- 让原本的后退按钮重新亮起来

但后续接受了一个现实：这个方向很可能会被浏览器底层沙盒和历史模型锁死。

### 6.2 阶段二：系统级 UI 覆盖 / System-Level UI Overlay

进一步提出了一个明显带有系统软件色彩的方案：

- 后台跟踪 Chrome 窗口位置
- 让本地软件在浏览器后退箭头区域上方覆盖一个自己的按钮
- 视觉上“盖住原本的”
- 同时这个覆盖按钮需要做到半透、可鼠标穿透，避免 Chrome 窗口失焦
- 平台能力上可以接受只支持 Windows

这已经不是普通插件思维，而是系统软件思维。

### 6.2.5 阶段 2.5：预加载页 pushState 占位 / Pre-Swap History Seeding

系统级 UI 覆盖方案是偏系统软件的重方案。在走到那一步之前，还有一条更轻、纯浏览器层的路线值得先试：

- 预加载页在被放入用户前台窗口之前
- 先用 `history.pushState` 往它自己的历史栈里塞一个占位 entry
- 占位 entry 指向用户点击前停留的 source 页面
- swap 完成后，用户按后退键就能落回 source

这条路线的优点：

- 完全在浏览器沙盒内，不需要本地软件干预窗口
- 跨平台天然可用
- 不依赖窗口坐标，不受 DPI / 多显示器影响

它的代价：

- pushState 只能占位，不能真正还原 source 的完整渲染状态
- 如果 source 依赖的内存状态已被 swap 丢弃，回到 source 可能会触发重新加载
- 对于期望"后退 = 瞬间恢复"的用户，体感略逊于历史嫁接

因此这条路线的定位是：

- 比后退变灰好
- 比系统级覆盖方案轻
- 可以作为阶段 2 系统覆盖方案真正落地前的默认兜底

### 6.3 阶段三：终局取舍 / Final Trade-Off

最终态度很明确：

- 哪怕这个功能最终没有做成
- 用户只能靠鼠标侧键完成前进后退
- 也不改变项目核心价值

因为只要守住“无加载、桌面软件般丝滑”这个核心，这点 UI 瑕疵仍然是可以接受的。

## 7. 项目分层原则 / Project Layering Principles

随着想法不断扩展，项目边界已经逐渐清晰成三层。

### 7.1 主项目主线 / Mainline

主项目对外应保持干净，重点放在：

- 低延迟网页交互
- 自适应预加载
- 行为树预测
- 资源感知调度
- 隐藏页热启动
- 白名单站点优化
- 桌面软件 + 插件的混合体系

这一层定义项目身份。

### 7.2 实验功能层 / Experimental Layer

“很酷但是有点危险”的功能，不应该删除，而是用来撑上限。它们的作用是让极客和开发者看到项目的潜力，而不是影响主项目声誉。

因此这些功能应该：

- 默认关闭
- 强制弹警告
- 手动开启
- 最好按站点单独启用
- 不出现在主 README 首页 headline
- 不作为主项目的第一印象
- 可以放在 `labs/`、`experimental/`、`research/` 等区域

### 7.3 独立研究层 / Isolated Research Layer

凡是涉及以下能力：

- 自动授权
- 密码管理器深度协同
- 后台推进认证步骤
- 容易被理解为“绕过正常交互”的能力

都不应该成为面向大众的默认主路径。它们可以存在于：

- 研究分支
- 单独实验区
- 私有分支
- 或只保留接口，不主推实现

## 8. 开源、发布与声誉策略 / Open Source, Release, and Reputation Strategy

当前公开策略已经比较明确。

### 8.1 完全开源免费 / Fully Open Source and Free

这个项目不走轻付费小服务路线。当前目标不是精细商业化，而是做出成果，用公开可验证的作品证明能力。

因此项目可以：

- 完全开源
- 完全免费
- 不做复杂付费墙
- 不搞订阅系统
- 不搞繁琐的商业化流程

### 8.2 捐赠优先 / Donation-First

如果需要支持入口，就走最轻量方式：

- 纯资源捐赠
- 二维码放在那里
- 让愿意支持的人自然支持
- 不让支付流程反过来拖累项目推进

### 8.3 GitHub 源码公开，但不发正式安装包 / GitHub Source Public, No Official Release Package

当前公开策略是：

- 代码可以开源到 GitHub
- 但不会提供正式发布包
- 不做面向普通用户的一键安装体验
- 不做公开 release 包
- 不主动降低门槛让一般人直接下载即用

这并不是“绝对没人拿得到”，而是保持声誉边界：

- 普通人不会轻易装上
- 如果有人自行下载源码、自己打包、自己乱用，并触发某些网站风控
- 这不属于主项目推荐用法，也不应反过来伤害项目声誉

### 8.4 风险声明的真实作用 / The Real Role of Disclaimers

这部分可以压缩成一句话：

- 风险功能可以存在
- 可以明确警告
- 可以声明因乱用导致网站封控等损失，软件概不负责
- 但这类功能的作用是展示 ceiling，而不是定义项目

高风险功能用来证明”能做到多远”，不是用来决定”别人第一眼怎么看你”。

### 8.5 隐私与本地数据边界 / Privacy and Local Data Boundary

完整的跳转消息长期保存在用户本地，这是本项目的一项关键隐私边界。相关原则：

- 所有原始跳转消息只保存在本地磁盘，不上传任何远端服务器
- 数据按 Chrome profile 隔离，不跨 profile 合并
- 不同用户账号 / 不同浏览器 profile 拥有各自独立的图
- 本地 app 只在机器内部提供系统能力接口，不承载远端传输逻辑
- AI 关键词生成也只在本机模型内完成，prompt 与推理结果不出机器

本地存储是否加密、是否与操作系统账号绑定，属于后续可选增强：

- 基础版本接受”本地磁盘可读”的假设
- 对安全敏感用户可后续提供”扩展启动时需解锁”的加固入口

这条边界不属于实验层，也不属于研究层，它是主项目身份的一部分。后续 README 与发布文案中应显式声明”所有行为数据只保留在本地”。

## 9. 后续性能扩展方向 / Future Performance Extensions

项目已经不再局限于“网页预加载”本身，而是逐渐朝系统层资源管理推进。

### 9.1 遮挡感知与深度休眠 / Occlusion-Aware Deep Hibernation

一个明确的后续扩展方向是：

- 如果浏览器长时间被其他全屏或无边框窗口盖住
- 借助本地软件更高的权限
- 可以尝试把整个 Chrome 压到极深度休眠状态
- 让它尽可能不再占用 CPU、GPU 显存和系统内存

虽然对外不一定会这样表述，但内部的理想逻辑很清楚：在某些情形下，后台浏览器就应该像“被卸载掉”一样安静。

### 9.2 游戏压力模式 / Game Pressure Mode

这个方向进一步具体化到了游戏场景：

- 如果系统能读取到游戏列表，或者至少知道某游戏已启动
- 游戏运行一段稳定时间后
- 就直接进入一个极端的游戏压力模式
- 目标是让整个 Chrome 进程在后台尽可能深地退出活跃资源竞争

直觉依据也很明确：

- 一般情况下，玩游戏时后台最容易抢占 CPU、GPU 显存和系统内存的，通常就是浏览器
- 因此如果用户进入游戏阶段，浏览器应该成为首要打压对象

这部分属于：

- 激进模式
- 平台能力上限展示
- 偏系统级资源回收
- 很适合让极客与开发者看到“这个项目不只是会网页小技巧”

## 10. 项目身份，不是什么 / What This Project Is Not

反过来归纳，项目不是：

- 不是普通的页面预取插件
- 不是只会在前端做小修小补的扩展
- 不是以收费为核心的轻量 SaaS
- 不是靠危险能力博眼球的灰色工具
- 不是把高风险功能当 headline 的项目

它真正要成为的是：

- 一套以“桌面级无加载网页体验”为目标的开源混合系统
- 主体是干净的性能与交互优化
- 上层保留非常强的实验空间
- 高风险能力只负责撑上限，不负责定义声誉

## 11. 一句总括 / One-Sentence Summary

这是一个由本地桌面软件与浏览器插件共同驱动的“零延迟 Web 系统”：它通过硬件感知、自适应阈值、点击链行为树、隐藏页热启动和跨窗口标签页置换，把高频网页流程推向桌面软件级的丝滑体验；同时保留一层默认关闭的激进实验能力，用来撑技术上限、展示系统想象力，但不让这些高风险部分反过来定义项目声誉。
