# 站外导航双层频数落地方案

这份文档只回答一个问题：

- 新的“站点频数 + 外站到站内子页面频数 + 站内频数”体系，如何在当前代码结构里落地，而不把现有预测链写成新的屎山。

这份文档不重复算法定义。

- 设计口径见：
  - [站外导航站内子页面槽位分配设计](./站外导航站内子页面槽位分配设计.md)
- 槽位分配算法代码见：
  - [站外导航站内子页面分配算法](./站外导航站内子页面分配算法)

---

## 1. 目标

当前外站导航场景，后续要从“单层候选列表直接按页级分数排序”切到：

1. 先按站点聚类
2. 再算站点级权重
3. 再做站点选择截断
4. 再在已选站点里分配页面槽位
5. 最后由站点内部子页面去抢这些槽位

同时保持：

- 站内导航逻辑不变
- preload 执行层不变
- 最终仍然输出一组具体页面候选给现有 runtime 去执行

换句话说，这次重构只改：

- 频数查询结构
- 预测编排结构
- 分数计算分层

不改：

- hidden-tab / prerender / prefetch 执行器
- 点击激活链
- 后台窗口模型

---

## 2. 必须先固定的边界

### 2.1 站点选择上限和页面槽位上限不是一个值

当前必须明确有四套独立约束：

1. `siteSelectionLimit`
   - 原生预加载组本轮最多允许多少个站点进入已选集合
2. `tabSiteSelectionLimit`
   - 真实标签页预加载组本轮最多允许多少个站点进入已选集合
3. `nativePageSlotLimit`
   - 原生预加载组本轮总共允许预加载多少个页面
4. `tabPageSlotLimit`
   - 真实标签页组本轮总共允许预加载多少个页面

这四套约束独立存在，不再混用。

### 2.2 参与槽位分配的站点集合必须先收敛

`allocateSelectedWithMinOne(...)` 的隐含前提是：

- `a >= n`

其中：

- `a = 当前分组自己的 pageSlotLimit`
- `n = 已选站点集合大小`

因此真实实现里，不能先拿所有站点去分配页面槽位。

必须先做：

- 站点聚类
- 站点权重排序
- 站点截断

当前推荐固定：

`effectiveSelectedSiteCount(group) = min(groupSiteSelectionLimit, groupPageSlotLimit, eligibleSiteCountInGroup)`

也就是说：

- 如果某个分组的页面槽位总数只有 `5`
- 那么某一组里就不能同时选出 `8` 个站点进入最终分配

否则算法语义会直接失效。

### 2.3 原有“默认子页面继承站点值”机制彻底删除

在新方案里：

- 站点值只在站点层比较
- 子页面值只在站点内部比较

因此旧逻辑里任何：

- 站点值直接灌给某个默认子页面
- page-level 缺失时拿 site-level 直接替代 page-level

都应该删除。

## 3. 频数模型怎么拆

当前后续应固定成 3 套频数：

### 3.1 `siteTransitionCount`

含义：

- 当前 source 页面来自外站
- 该 source 页面导航到某个目标站点的频数

用途：

- 只用于外站场景下的站点级权重

### 3.2 `outboundPageTransitionCount`

含义：

- 当前 source 页面来自外站
- 该 source 页面导航到某个目标站点内具体子页面的频数

用途：

- 只用于外站场景下，站点内部子页面排序

### 3.3 `intraSitePageTransitionCount`

含义：

- source 和 target 都在同一站点内部
- 某个站内页面跳到另一个站内页面的频数

用途：

- 只用于站内场景排序

### 3.4 不再保留一个模糊的 `transitionCount`

当前实现里最容易继续污染逻辑的地方，就是“不同语义的频数最后又收敛成一个 `transitionCount`”。

新方案里必须避免这一点。

也就是说：

- Rust 查询层
- JS metrics enrich 层
- scoring 层

都不应继续只传一个模糊 `transitionCount`。

应显式传：

- `siteTransitionCount`
- `outboundPageTransitionCount`
- `intraSitePageTransitionCount`

最终由不同场景自己决定吃哪一套。

---

## 4. Rust 查询层怎么改

当前最稳的方案不是重新发明很多 query 类型，而是在现有：

- `get-candidate-transition-metrics-batch`

上扩展返回字段。

### 4.1 现有接口保留

先保留这个批量 query 入口不变，避免把 JS 管线和 Wasm 桥一起炸掉。

输入仍然可以是：

- `sourceNodeId`
- `sourcePageUrl`
- `candidates[]`

### 4.2 返回结构升级

每个 candidate 现在建议返回：

- `url`
- `targetNodeId`
- `targetPageUrl`
- `siteTransitionCount`
- `outboundPageTransitionCount`
- `intraSitePageTransitionCount`

其中：

- `outboundPageTransitionCount`
  - 就是当前现有 page-level external query 的那套值
- `intraSitePageTransitionCount`
  - 当 source/target 属于同站时，从现有 page-transition buckets 查
- `siteTransitionCount`
  - 继续从 site-level buckets 查

### 4.3 不在 Rust 查询层做站点聚类

站点聚类建议仍放在 JS 层。

原因：

- 当前候选链接信息本来就由 content script 提供，并先进入 JS
- JS 更适合把候选按 `targetNodeId` 聚类
- Rust 保持做“批量查询频数数据”，不要把编排逻辑也塞进去

也就是说：

- Rust 负责“查值”
- JS 负责“组站点 / 算流程”

---

## 5. JS 预测管线怎么拆段

当前最关键的是不要把新方案直接糊回 `candidate-pool.js` 或 `metrics.js`。

建议后续按下面 4 段来落。

### 5.1 第一段：原始候选池

文件仍可从：

- `background/preload/prediction/candidate-pool.js`

输出，但这一步只负责：

- 当前页面真实链接去重
- 提取 link info
- 规范化 URL / nodeId / targetPageUrl

不在这里做站点分组和最终排序。

输出结构先叫：

- `rawCandidatePool`

### 5.2 第二段：频数 enrich

由：

- `background/preload/prediction/metrics.js`

负责，把 Rust 批量 query 回来的 3 套频数挂到每个 candidate 上。

这一步输出应是：

- `candidateMetricPool`

其中每个 candidate 至少有：

- `siteTransitionCount`
- `outboundPageTransitionCount`
- `intraSitePageTransitionCount`

### 5.3 第三段：站点聚类和站点入选

建议新增一层：

- `background/preload/prediction/site-selection.js`

它只负责：

1. 先按执行策略拆成原生预加载组和真实标签页组
2. 在组内把候选按 `targetNodeId` 聚类
3. 计算每个站点的：
   - `siteWeight`
   - `cap`
4. 按 `siteWeight` 排序
5. 用当前组自己的站点上限截断
6. 计算 `effectiveSelectedSiteCount`
7. 调槽位分配算法，得到每个站点拿多少个页面槽位

注意：

- `siteWeight` 只吃：
  - 站点频数乘区
  - 站点 AI 乘区
- `cap` 只来自当前页面实际候选链接中，该站点可导航子页面数量

### 5.4 第四段：站点内部页面选择

建议新增：

- `background/preload/prediction/page-selection.js`

它只负责：

- 对每个已选站点内部的候选子页面计算 `pageWeight`
- 再按分到的槽位数挑出前 `k` 个页面

外站场景：

- `pageWeight` 吃：
  - `outboundPageTransitionCount`
  - 子页面 AI 乘区

站内场景：

- `pageWeight` 吃：
  - `intraSitePageTransitionCount`
  - 子页面 AI 乘区

### 5.5 最后才回到 strategy router

只有当“最终页面候选列表”确定之后，才交给：

- `strategy-router.js`

去决定：

- hidden-tab
- prerender
- prefetch

这样可以避免“策略分流”和“频数系统重构”互相污染。

## 6. AI 匹配怎么接

### 6.1 站点级 AI

当前 interest keywords 生成链不需要大改。

后续只要新增一层：

- `buildSiteAiKeywordMatch(...)`

输入：

- `interestKeywords`
- 站点语义包

站点语义包建议最开始只包含：

- 该站点在当前页面所有候选链接的文本集合
- 该站点目标页历史关键词的并集

输出：

- `siteAiMultiplier`

### 6.2 子页面级 AI

保留现有第一阶段思路：

- `interestKeywords`
  匹配
- `candidateSemanticBundle = linkInfo + targetPageKeywords`

输出：

- `pageAiMultiplier`

### 6.3 两层不能互相替代

后续实现里不要偷懒把：

- 子页面 AI 最大值

直接拿来当站点 AI。

站点 AI 是站点层信号，子页面 AI 是页级信号，两层必须分开算。

## 7. 建议的最小迁移顺序

为了避免把现有链路一次性打爆，建议按下面顺序做。

### 第一步：先改 query 返回结构

- Rust query 扩展成 3 套频数字段
- JS metrics 只先接字段，不改排序逻辑

完成标准：

- 当前 candidate 上已经同时能看到：
  - `siteTransitionCount`
  - `outboundPageTransitionCount`
  - `intraSitePageTransitionCount`

### 第二步：引入站点聚类层

- 新增 `site-selection.js`
- 先只做：
  - 按执行策略分组
  - 组内聚类
  - `cap`
  - `siteWeight`
  - 各组自己的站点上限截断

完成标准：

- 还不真正改最终排序
- 但 debug 里能看到“本轮已选站点集合”

### 第三步：接入页面槽位分配

- 把 `allocateSelectedWithMinOne(...)` 接进站点层
- 让 debug 能看到：
  - 每个已选站点拿到多少页面槽位

完成标准：

- 还不动 runtime 执行层
- 但最终候选页列表已经变成“按站点先分槽，再站内选页”

### 第四步：切正式输出

- 用新的最终页面候选列表替换旧 `selectedTargets`
- 然后再交给现有 strategy router

完成标准：

- runtime 执行器保持不变
- 但输入它的页面集合已经换成新频数系统产物

---

## 8. 当前不建议做的事

为了避免再长一层屎山，当前先不要做这些：

- 不要在 `metrics.js` 里直接把站点聚类、站点分槽、页面排序都写进去
- 不要在 Rust query 层直接做站点入选和页面分槽
- 不要重新引入“默认子页面继承站点值”
- 不要把站点级 AI 和子页面级 AI 合并成一层
- 不要一边改频数系统，一边同时重写 strategy router

---

## 9. 一句话落地原则

这次重构应遵守一条原则：

**先把“值查出来”，再把“站点选出来”，再把“槽位分出来”，最后才把“页面挑出来”。**

顺序不能乱。乱了就会重新回到“各种语义全糊成一个 transitionCount”的旧路上。
