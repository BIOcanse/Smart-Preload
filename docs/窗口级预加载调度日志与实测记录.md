### Chromium/Edge/app 预加载隔离 smoke

```powershell
node scripts\testing\preload-browser-isolation-smoke.mjs
```

测试方式：

- 同时拉起 Playwright bundled Chromium 和真实 Microsoft Edge。
- 每个浏览器使用独立临时 profile 和独立 unpacked extension 副本。
- 本地 app 以 `--host` 启动，并临时把本次测试扩展 origin 写入 staging app 的 allowed origins；测试结束后恢复原文件。
- 测试页面全部由本地 HTTP server 提供，使用 `.test` 域名映射到 `127.0.0.1`，不访问真实站点。
- 每个浏览器页面同时提供同源 native 候选和跨站 hidden-tab 候选。

结果摘要：

```text
ok: true
appHealth.ok: true
browserCount: 2
chromium hiddenCount: 5, prerenderCount: 5, overlap: 0, hiddenBySystem: true
edge hiddenCount: 5, prerenderCount: 5, overlap: 0, hiddenBySystem: true
```

观察：

- hidden-tab 和 native preload 的 URL 集合没有交叉，说明两套状态没有混淆。
- 两个浏览器实例同时在线时，各自扩展 origin、source window、preload window、hidden entries 和 native entries 独立。
- 本地 app 成功处理 `native-app.registration.success`、`native-app.windows.hide.result`、`preload-window.hide.system-success`。
- `chrome.windows.get(preloadWindowId).state` 对 Win32 system hide 仍可能返回 `normal`，所以判断后台窗口是否被系统隐藏应以 app 返回的 `hwnd`、`hiddenBySystem=true` 和 app hide 事件为准，而不是只看 Chrome API 的 minimized 状态。

## 设计符合性检查

### 符合项

1. 预加载不再只依赖当前标签页：候选快照按 source tab 保存在 `scheduler.candidateSelectionSnapshotsByTabId`。
2. 上层重排使用 `scoreSignals` 和 `attentionPool`，没有重新跑候选评分。
3. 分片默认值是 60 秒，且设置页已有可编辑字段。
4. pending 余额按 `tabId + normalizedPageUrl` 保留，未满分片不会丢失。
5. 点击拦截 smoke 未出现吞输入：被拦截的 5 次全部激活命中，未拦截的场景正常导航。

### 发现的问题与处理

1. 原先调度日志不够细，只能看到最终选择，缺少 `scoreSum / linkValueMultiplier / dwellShare / rawSlots` 等中间量。已补齐。
2. 短 smoke 在默认 60 秒分片下不会验证真实浏览器里的完整分片入池事件。这不是设计不符，而是测试时长不足；当前用单测覆盖该路径，后续如需浏览器级覆盖，可单独设置较短测试分片运行。
3. 10 次点击 smoke 中后 4 次没有预加载但正常导航。这不违反当前设计：全局调度允许部分低权重或未分配额度的 tab 不预加载；关键是不能吞点击，结果为 0 次吞点击。
4. 复查 `scoreSum` 路径时发现：上层 `scoreSignals` 已正确保存，但 scheduler 首次登记候选时如果 `selection` 为空，`candidateSelectionSnapshotsByTabId[tabId].selectedTargets` 可能为空，导致后续 stored snapshot 轻量重排有分数但没有下层目标排行。已修复为：首次登记时用已算好的 `scoredCandidatePool` 生成宽选目标快照，再保存 `selectedTargets`；这一步不重新跑候选评分。

## scoreSum 数值复查

当前实现的汇总边界：

```text
content script 可见可导航链接，最多 40 个
  + Google bookmark 候选
  -> URL 去重
  -> filterSourceSpecificCandidatePool
  -> enrich metrics
  -> scorePreloadCandidatePool
  -> buildPreloadSchedulerScoreSignals
```

`buildPreloadSchedulerScoreSignals` 在下层规则过滤和 site selection 之前运行，因此它汇总的是进入 scored candidate pool 的所有候选，而不是最终被截断的目标。

按系统分组后：

```text
scoreSignals.native.scoreSum = sum(native candidate.score ^ 1.5)
scoreSignals.tab.scoreSum = sum(hidden-tab candidate.score ^ 1.5)
linkValueMultiplier = buildSchedulerLinkValueMultiplier(scoreSum)
                    = 1 + ln(1 + scoreSum)
```

这里的 `score^1.5` 只用于上层标签页调度的汇总信号，不改变标签页内部链接排序本身。目的：保留“高质量候选链接”的优势，减少大量 `score=1` 的低价值链接靠数量以次充好。`linkValueMultiplier` 也只用于上层资源分配，不参与后续候选分数继续计算，因此改为无固定上界的对数函数。

典型乘数：

```text
scoreSum 1    -> 1.6931
scoreSum 5    -> 2.7918
scoreSum 10   -> 3.3979
scoreSum 40   -> 4.7136
scoreSum 100  -> 5.6151
scoreSum 500  -> 7.2166
scoreSum 1000 -> 7.9088
```

对比示例：

```text
10 个 score=1 的候选 -> scoreSum 10.0000 -> multiplier 3.3979
1 个 score=10 的候选 -> scoreSum 31.6228 -> multiplier 4.4850
```

判断：乘数不再有固定上界，但增长速度是对数级。由于候选链接最多 40 个，普通页面的 `scoreSum` 多数会落在几十到几百区间，对应乘数约 4.7 到 6.x；高分候选页面会比纯数量页面更容易拿到上层额度，且优势不会被旧的 3 倍上限过早压平。这符合“上层表达该 tab 内部候选总体价值，同时避免低分链接数量线性吞掉全部额度”的设计。

本轮没有发现会阻止继续发布准备的设计违背项。需要继续观察真实用户日志中的两类事件：
